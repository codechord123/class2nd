# 2학기 학급 자치 시스템 — 리뉴얼 설계안 (v1)

> 목적: 코딩 착수 전, **① 배포/스택 방향 · ② 데이터베이스 선택 · ③ 디렉토리 구조 · ④ 데이터 스키마 · ⑤ DB 읽기/쓰기 최적화 전략**을 확정한다.
> 최우선 과제는 1학기의 **Firebase 읽기/쓰기 한계치 초과 문제**를 구조적으로 없애는 것.

---

## 0. 결정 사항 요약 (먼저 볼 것)

| 항목 | 1학기 | 2학기 제안 | 이유 |
|---|---|---|---|
| 배포 | Apps Script (수동 배포) | **Vercel + GitHub 자동 배포** | git push = 배포. Cron·서버함수 사용 가능 |
| 앱 구조 | 단일 HTML(3천 줄) | **Next.js(App Router) + 컴포넌트 분리** | 리렌더 격리, 유지보수, 서버 집계 |
| 상태관리 | 거대 useState 뭉치 | **Zustand(UI) + TanStack Query(서버데이터)** | 캐시·dedup으로 읽기 폭증 차단 |
| DB | Firestore(문서당 과금) | **택1: Supabase(추천) / Firestore 유지** | §2 참조 |
| 집계 방식 | 매 접속마다 전체 로드 | **서버 사전집계(Cron) → 결과만 읽기** | 읽기량 수십분의 1로 |
| 1학기 정적데이터 | (해당없음) | **GitHub JSON → 빌드 번들 / CDN fetch** | DB 안 씀 |

핵심 아이디어 한 줄: **"원시 평가데이터를 클라이언트가 모두 읽어 계산" → "서버가 하루 1번 계산해 저장, 클라이언트는 결과 몇 줄만 읽기".**

## 1. 배포 & 기술 스택

### 1.1 배포 — Vercel + GitHub
- `main` 브랜치 push → 프로덕션 자동 배포, PR → 프리뷰 배포.
- **Vercel Cron** 으로 스케줄 작업 실행 (1학기의 수동/Apps Script 부재를 대체):
  - 매일 저녁 종회 후 → 일일 점수·모둠 랭킹 집계
  - 격주 금요일 16:00 → MVP·코인 정산
- **환경변수**로 DB 시크릿 관리 (`.env` → Vercel 대시보드). firebaseConfig/서비스키를 소스에 하드코딩하지 않음(1학기와 차이).

### 1.2 프레임워크 — Next.js (App Router) + TypeScript + Tailwind
- **왜 Next.js**: Vercel 네이티브 + **API Route(서버 함수)** 가 최적화의 핵심. 클라이언트가 DB를 직접 난사하지 않고 서버 함수가 배치조회·캐시·집계를 대행 → DB 히트 최소화.
- **왜 TypeScript**: 1학기의 "기록 저장/불러오기 에러(오차)" 재발 방지. 스키마 타입으로 컴파일 단계에서 필드 실수 차단.
- **대안**: 단순 Vite SPA도 가능하나 서버 함수/Cron이 없어 최적화 레버가 약함 → 이번 최우선 과제상 Next.js 권장.

### 1.3 상태관리
- **TanStack Query (React Query)** = 서버 데이터 캐시 계층. `staleTime`을 길게(예: 자리표 1일, 명단 무한) 잡아 **같은 데이터 재조회를 자동 차단**. 이것이 읽기 절감의 1등 공신.
- **Zustand** = UI/세션 상태(현재 탭, 로그인 유저, 선택된 날짜 등)만. 가볍고 리렌더 격리.

## 2. 데이터베이스 선택 (고민 지점)

두 안을 제시하되, **이 워크로드에는 Supabase를 추천**한다. 최종 선택은 선생님이.

### 안 A — Supabase (Postgres) · 추천
| 장점 | 설명 |
|---|---|
| 과금 모델이 문제를 없앰 | Firestore는 **문서 읽기 건수**로 과금·제한 → 1학기 한계 초과의 근본 원인. Postgres는 행/저장/대역폭 기준이라 "읽기 5만 회" 개념 자체가 사라짐 |
| SQL 집계 1방 | 랭킹·합계·MVP를 `GROUP BY`·window 함수로 **한 쿼리**에 계산. 25×25 평가행도 서버에서 즉시 집계 |
| 관계형이 딱 맞음 | students·groups·schedules·evaluations·scores는 전형적 관계 데이터 |
| RLS | Row Level Security = Firestore 규칙과 동등한 권한제어 |
| 무료 한도 넉넉 | 25명 규모는 무료 티어로 충분 |
| 실시간은 옵션 | 필요한 곳만 realtime 구독 |

단점: 선생님이 1학기에 Firebase에 익숙 → 새 학습곡선. Auth를 Supabase Auth로 재구성 필요.

### 안 B — Firebase Firestore 유지
| 장점 | 단점 |
|---|---|
| 1학기 경험·Auth·보안규칙 재사용 | 문서당 읽기 과금 = 한계 초과 원인 그대로 |
| 인수인계 문서(③) 스키마 그대로 | 집계 시 다수 문서 읽기 → 여전히 비쌈 |

→ **B를 택해도** §5의 "서버 사전집계 + React Query 캐시 + 정적분리"를 적용하면 한계 초과는 막을 수 있음. 다만 A가 구조적으로 더 안전.

> **결정 필요 ①**: A(Supabase) / B(Firestore 유지) — 아래 스키마는 두 안 모두 병기.

## 4. 데이터 스키마

### 4.1 개념 모델(공통)
- `students` 25명: 이름·성별·의장여부·소속모둠(의장)·1학기재화는 **정적**
- `schedules` 21주: 주차별 5모둠×(의장+4역할) 배치 + 수동보정
- `evaluations` 모둠 내: 평가자→대상 +1/0/-1
- `groupVotes` 모둠 간: 평가자→모둠 +1/0/-1
- `dailyScores` 일일 집계(서버가 씀)
- `biweeklyScores` 격주 MVP(서버가 씀)
- `coinTxns` 코인 원장

### 4.2 Supabase(Postgres) 스키마 — 안 A
```sql
students(id pk, name, gender, is_chair bool, chair_group int, is_wildcard bool)
schedules(week int pk, week_start date, data jsonb, manual_overrides jsonb)
   -- data: [{groupId, chair, members:[{studentId, role}]}]
evaluations(date date, evaluator_id, target_id, score int2,   pk(date,evaluator_id,target_id))
group_votes(date date, evaluator_id, group_id int, score int2, pk(date,evaluator_id,group_id))
daily_scores(date date, student_id, peer int, group_rank int, bonus int, total int, pk(date,student_id))
biweekly_scores(period int, student_id, sum int, is_mvp bool, mvp_rank int, pk(period,student_id))
coin_txns(id pk, student_id, type text, amount int, date date, note text)
student_auth(student_id pk, hash text, updated_at)
seat_change_requests(id pk, student_id, week int, target_group int, target_role text, created_at, status)
```
집계는 SQL 뷰/함수로: 예) 일일 모둠내 점수 = `SELECT target_id, SUM(score) FROM evaluations WHERE date=$1 GROUP BY target_id`.

### 4.3 Firestore 스키마 — 안 B (③ 인수인계 문서 유지)
```
students/{id}           name,gender,isChair,chairOfGroup,isWildcard   (1학기재화는 정적)
schedules/{1..21}       weekStart, groups[], manualOverrides[]
evaluations/{date}/{evaluatorId}   { targetId:+1|0|-1, ... }
groupVotes/{date}/{evaluatorId}    { groupId:+1|0|-1, ... }
dailyScores/{date}/{studentId}     peer, groupRank, bonus, total   ← 서버가 씀
biweeklyScores/{period}/{studentId} sum, isMVP, mvpRank            ← 서버가 씀
coinTxns/{id}                      studentId,type,amount,date,note
studentAuth/{studentId}            hash, updatedAt
_sessions/{id}                     활동로그
```
> 집계 문서를 별도로 두는 게 핵심: 클라이언트는 `dailyScores`·`biweeklyScores`만 읽고, 원시 `evaluations` 전체를 읽지 않음.

## 5. DB 읽기/쓰기 최적화 전략 ★ (최우선 과제)

1학기 문제 = **"접속·기록·호출 때마다 전체 데이터 로드"**. 아래 7원칙으로 구조적으로 차단.

### 원칙 1 — 전체 컬렉션 로드 절대 금지
- 탭 진입 시 그 탭에 필요한 문서만. 앱 시작 시 전체 스캔 없음.
- `loadedRef` 가드 대신 **React Query 캐시 키**로 자동 1회 로드.

### 원칙 2 — 서버 사전집계(Precomputation)  ← 가장 큰 절감
- 원시 평가(evaluations·groupVotes)는 25×25까지 커짐 → 클라이언트가 읽어 계산하면 폭증.
- **Vercel Cron이 하루 1번 서버에서 집계** → `dailyScores`/`daily_scores`에 학생당 1행 저장.
- 학생/학부모/교사는 **집계 결과(학생당 1개)만 읽음**. 원시행은 안 읽음.
- 예상: 학생 25명 × 하루 10접속 × 읽는 집계문서 ~5 = **~1,250 reads/day** (Firestore 무료 5만의 2.5%). 1학기엔 접속 1회에 수백 문서를 읽어 금세 초과.

### 원칙 3 — 클라이언트 캐시 & 낙관적 업데이트
- React Query `staleTime`: 명단/자리표=길게(수 시간~1일), 점수=집계 주기와 동기.
- **본인이 쓴 뒤엔 재조회 없이 로컬 캐시만 갱신**(낙관적 업데이트) — 1학기 "쓰고 나서 전체 다시 읽기" 제거.
- `localStorage` 버전드 캐시: 21주 자리표·명단 등 거의 불변 데이터는 로컬에서, 버전 바뀔 때만 재fetch.

### 원칙 4 — 정적 데이터는 DB에서 뺀다
- **1학기 거북이 기록·실버/학급 토큰** → `data/static/*.json` (GitHub) → 빌드 번들 or 최초 1회 CDN fetch. **Firebase 미접근**, 2학기 연산과 완전 격리.
- 21주 자리표도 사전계산 후 정적 JSON으로 둘 수 있음(옵션) → 읽기 0.

### 원칙 5 — 실시간(onSnapshot/realtime) 최소화
- 전체 컬렉션 실시간 구독 금지. **열람 중인 게시글 1개** 등 꼭 필요한 단일 문서만 구독.

### 원칙 6 — 페이지네이션/Limit
- 건의·투표 게시판은 최근 N개만(무한스크롤). 오래된 글은 요청 시에만.

### 원칙 7 — 읽기 프록시(서버 함수 캐시)
- `api/read/*`가 서버에서 배치조회 + `Cache-Control: s-maxage, stale-while-revalidate`.
- 여러 학생이 같은 "이번 주 자리표"를 요청해도 **DB 히트는 캐시가 흡수** → 실제 DB 읽기 대폭 감소(특히 Firestore에서 효과 큼).

### 학부모 대시보드 최적화
- 학부모는 **읽기 전용 + 자녀 1명분**만. 서버가 만든 `dailyScores`/`biweeklyScores` 집계행만 조회 → 매우 저렴.
- 자녀 매핑(parent→studentId)만 1회 로드 후 캐시.

## 6. 핵심 기능 알고리즘 (요구사항 반영)

- **Team(모둠 평가)**: 현재 주차 `schedule`에서 "우리 모둠 + 나머지 4모둠"만 렌더(전체 학급 X). 모둠 내 +1/0/-1 상호평가, 다른 모둠 버튼 → 해당 모둠 전원 각 1점. **점수는 학생 단위 누적**이라 모둠이 2주마다 바뀌어도 유지(모둠이 아니라 studentId에 귀속).
- **모둠 간 랭킹**: Dense Ranking → 개인점수(1위5 … 5위1), 같은 모둠 전원 동일점(③ §5.5). 서버 집계에서 계산.
- **자리배치**: 21주 시뮬레이티드 어닐링 사전계산(③ §5.1) → 저장. 날짜로 현재 주차 자동 전환. 토큰 신청 마감 **수요일 자정**, 동일 자리 **선착순**, 교사 수동 swap.
- **거북이 독서**: 주 3권 의무, 미달 권수 계산해 홈 상단 경고. 짜파게티 배너 고정 + 순위 **가로 캐러셀**.
- **상점**: 2학기 재화는 DB, 1학기 재화(실버·학급토큰)는 정적 JSON 표시만, 연산 격리.

## 8. 제안 진행 순서 (동의 시)

1. **Phase 0**: 스택 확정(위 결정 5개) + 저장소 스캐폴딩(Next.js) + 1학기 정적 JSON 커밋
2. **Phase 1**: 인프라 — 인증(교사/학생/학부모), DB 어댑터, 공통 UI/레이아웃
3. **Phase 2**: 자리배치 — 어닐링 사전계산 + 21주 캘린더 + 날짜 자동전환
